# Base de données

Postgres via Supabase. Schéma versionné dans `supabase/migrations/`, appliqué via `npx supabase db push` (voir [README.md](README.md)).

## ⚠️ Piège connu : dérive schéma réel ↔ migrations versionnées

Plusieurs changements de schéma en prod (des policies RLS notamment) ont été appliqués hors du flux `supabase db push` — probablement collés directement dans le SQL Editor du dashboard — et n'apparaissaient donc dans aucune migration du repo. Deux conséquences rencontrées en juillet 2026 :

1. **L'historique de migration de la CLI était désynchronisé.** `supabase db push` essayait de rejouer une migration déjà appliquée en prod (colonnes/tables `if not exists`, donc inoffensif) jusqu'à tomber sur un `create policy` sans garde, qui plantait tout le push. Fix : `supabase migration repair --status applied <version>` pour resynchroniser l'historique avant de repush.
2. **Des policies RLS non versionnées ont été trouvées en auditant la prod** (`supabase db query --linked "SELECT * FROM pg_policies WHERE ..."`) qui n'existaient dans aucun fichier de migration — dont une policy `ALL` pour le rôle `anon` sur `market_opportunities` (n'importe qui pouvait écrire/supprimer sans être connecté). Voir plus bas.

**Leçon : ne jamais modifier le schéma de prod directement dans le dashboard.** Toujours passer par une migration versionnée + `db push`, même pour un changement d'une ligne. Si un doute existe sur l'état réel de la prod, vérifier avec `npx supabase db query --linked "..."` (lecture seule, sûr) plutôt que de faire confiance aux fichiers de migration seuls.

**Même piège pour les Edge Functions** : une fonction `scan-market` est restée déployée sur Supabase jusqu'au 2026-07-11 sans jamais avoir été trackée dans ce dépôt Git. C'était le déploiement Supabase de l'ancien `api/scan-market.ts`, un « faux scanner » à données codées en dur, explicitement remplacé par le vrai scraper Playwright (`scripts/vinted-scan.ts`) dans le commit `c8b7905` (2026-07-07) — mais jamais supprimé côté Supabase quand son code source local a disparu. Aucune fausse donnée n'a jamais atteint la table `market_opportunities` (vérifié : les 309 lignes en prod correspondent toutes à de vraies URLs Vinted à identifiant numérique), mais la fonction restait appelable et aurait pu en insérer. Supprimée via `npx supabase functions delete scan-market`. **Leçon : `npx supabase functions deploy` déploie, mais rien ne supprime automatiquement une fonction quand son code source est retiré du dépôt — un `supabase functions list` périodique, recoupé avec les dossiers réellement trackés dans `supabase/functions/`, est le seul moyen fiable de détecter ce genre de dérive.**

## RPC (fonctions SECURITY DEFINER)

| Fonction | Rôle | Accès |
|---|---|---|
| `increment_usage(p_user_id, p_month)` | Incrémente le compteur d'analyses du mois | `authenticated` uniquement, vérifie `auth.uid() = p_user_id` en interne |
| `decrement_credit(p_user_id)` | Réserve un crédit IA (le décrémente). **Depuis le 2026-07-11 (P0.1) : lève l'exception `insufficient_credits` au lieu de "floor" silencieusement à 0** — permet à l'appelant (la fonction Edge `analyze-clothing`) de distinguer "crédit consommé" de "déjà à zéro" et de réserver *avant* l'appel Gemini plutôt qu'après | `authenticated` uniquement, vérifie `auth.uid() = p_user_id` en interne |
| `refund_credit(p_user_id)` | **Nouveau (P0.1)**, symétrique de `decrement_credit` — recrédite un crédit. Utilisé par `analyze-clothing` pour rembourser une réservation quand l'appel Gemini échoue après coup | `authenticated` uniquement, vérifie `auth.uid() = p_user_id` en interne |
| `set_default_vinted_account(target_account_id)` | **Phase B multi-comptes** (2026-07-09) : bascule le compte Vinted par défaut de l'utilisateur de façon atomique (désactive l'ancien, active le nouveau dans la même transaction). Un index partiel `unique (user_id) where is_default` sur `vinted_accounts` garantit qu'un seul compte par défaut peut exister même en cas d'appel concurrent. | `authenticated` uniquement, vérifie que le compte cible appartient à `auth.uid()` en interne |
| `allocate_next_sku(p_user_id)` | **Sprint extension V1** (2026-07-13) : calcule le plus petit entier positif non utilisé par une annonce active (`status <> 'vendu'` et `vinted_status` distinct de `'deleted'`) de l'utilisateur — une annonce vendue/supprimée libère implicitement son numéro sans jamais réécrire le `sku` déjà enregistré sur cette ligne. Verrou `pg_advisory_xact_lock(hashtext(p_user_id::text))` pour qu'un double insert concurrent ne calcule jamais le même numéro. Appelée par le trigger `trg_assign_sku_before_insert` sur `listings` (`BEFORE INSERT`, seulement si `NEW.sku is null` — un import qui reprend un numéro déjà présent dans le titre Vinted fournit son propre `sku` et court-circuite l'allocation) | `authenticated` uniquement, vérifie `auth.uid() = p_user_id` en interne |
| `handle_new_user()` | Trigger sur `auth.users`, crée le profil à l'inscription | Trigger uniquement, aucun accès RPC direct |

`decrement_credit`/`increment_usage` étaient à l'origine appelables par le rôle `anon` **sans aucune vérification d'identité** — n'importe qui, non connecté, pouvait vider les crédits de n'importe quel utilisateur en boucle en devinant/énumérant des UUID. Corrigé en juillet 2026 (migration `20260708143000_secure_credit_rpcs.sql`) : accès restreint à `authenticated`, vérification `auth.uid() = p_user_id` ajoutée dans le corps des deux fonctions, `search_path` fixé (bonne pratique Postgres pour les fonctions `SECURITY DEFINER`).

## Verrouillage serveur du plan / crédits / abonnement (P0.1, 2026-07-11)

L'audit pré-lancement a confirmé en direct (via `npx supabase db query --linked`, en simulant un rôle `authenticated` réel dans une transaction annulée) qu'un utilisateur pouvait exécuter `supabase.from('profiles').update({plan:'pro', credits:999999})` depuis la console du navigateur et que ça fonctionnait — RLS protège les **lignes**, pas les **colonnes**, et rien ne restreignait quelles colonnes `authenticated` pouvait écrire sur `profiles`.

**Principe retenu, à réutiliser pour toute future colonne sensible** : les privilèges Postgres au niveau colonne (`GRANT`/`REVOKE UPDATE (colonne) ON table ...`) s'ajoutent à la RLS, ils ne la remplacent pas. Une colonne "sensible" (dont la valeur ne doit jamais dépendre de la volonté du client) doit avoir son `UPDATE` révoqué pour `authenticated`, et n'être modifiable que par une fonction `SECURITY DEFINER` qui vérifie `auth.uid() = p_user_id` en interne (tourne avec les privilèges du propriétaire de la fonction, donc pas affectée par le `REVOKE`) — ou, une fois Stripe branché, par un webhook en `service_role`.

**Piège rencontré en l'appliquant** : `REVOKE UPDATE (colonne) ON table FROM authenticated` **ne fait rien** si le privilège avait été accordé au niveau de la **table entière** (`GRANT UPDATE ON table TO authenticated`, sans liste de colonnes) — Postgres stocke les ACL de table (`relacl`) et de colonne (`attacl`) séparément, et un `REVOKE` colonne par colonne ne peut pas retirer un droit accordé au niveau table. `information_schema.column_privileges` affiche quand même la colonne comme "autorisée" dans ce cas (elle combine les deux ACL), ce qui peut faire croire à tort qu'un `REVOKE` colonne a fonctionné. Repéré sur `usage.analyses_count` (corrigé par `20260711091500_fix_usage_table_level_grant.sql`, un `REVOKE UPDATE` complet au niveau table) après que le premier essai colonne par colonne (`20260711090000_lock_billing_columns.sql`) n'ait silencieusement rien changé — vérifié après coup via `information_schema.table_privileges` (qui, lui, distingue vraiment table vs colonne). **Toujours vérifier via une vraie tentative de contournement (`SET LOCAL ROLE authenticated; SET LOCAL request.jwt.claims = '{"sub":"..."}'; UPDATE ...; ROLLBACK;`) plutôt que de faire confiance à `information_schema.column_privileges` seul.**

`profiles` (colonne `full_name` seule autorisée) et `subscriptions` (entièrement lecture seule pour le propriétaire) ont utilisé un `REVOKE UPDATE ON table FROM authenticated` complet suivi d'un `GRANT` ciblé sur les colonnes légitimes — ce schéma fonctionne du premier coup, contrairement au `REVOKE` colonne par colonne isolé.

`analyze-clothing` (fonction Edge) applique désormais le quota côté serveur de façon atomique : réservation (`decrement_credit`) **avant** l'appel Gemini si le plan est `free`, remboursement (`refund_credit`) si Gemini échoue après coup, incrémentation (`increment_usage`) seulement après succès. Le client (`GeneratorPage.tsx`) ne fait plus aucun appel RPC de crédit lui-même — il se contente de rafraîchir le profil après une génération réussie.

## Storage

Bucket `listing-images` (photos des annonces). **Créé hors du flux `db push` avant le 2026-07-13** (aucune migration versionnée ne le créait, exactement le piège documenté en tête de ce fichier) **et jamais réellement utilisé par du code** avant cette date — `GeneratorPage.tsx` enregistrait directement les `blob:` URL du navigateur dans `listings.image_urls`, qui cassaient dès la fermeture de l'onglet d'origine. `supabase/migrations/20260713100000_add_listing_photos_bucket.sql` versionne enfin sa création (`on conflict do nothing`, idempotent que le bucket existe déjà ou non) et ajoute des policies `insert`/`update`/`delete` scopées par dossier `{user_id}/...` ; `src/lib/storage.ts::uploadListingPhotos()` y écrit réellement désormais.

**Corrigé le 2026-07-23** (`20260723130000_remove_listing_images_public_list_policy.sql`) : la policy `listing_images_public_read` (`select` accordé à `public` sans restriction de ligne) permettait de **lister** tous les fichiers du bucket, pas seulement d'y accéder par URL connue, signalé par `supabase db advisors`. Supprimée sans casser l'affichage réel : le bucket reste `public = true`, et `getPublicUrl()` (seule méthode utilisée par `src/lib/storage.ts`, confirmé par grep sur `src/`/`extension/` — aucun `.list()` ni `.download()`) contourne entièrement les RLS de `storage.objects` pour un bucket public.

## Vérifier l'état réel de la prod

```bash
npx supabase link --project-ref <ref>
npx supabase db query --linked "SELECT * FROM pg_policies WHERE tablename = '...';"
npx supabase db advisors --linked --type all   # audit securite/perf integre Supabase
```

`db advisors` a été la source de la plupart des trouvailles de sécurité de juillet 2026 (RLS manquantes, fonctions sans `search_path`, policies dupliquées, bucket storage trop permissif). À relancer périodiquement, surtout après tout ajout de table ou de RPC.
